Audiobook pipeline shape

Three stages: chunk the source, loop synthesis, concat with ffmpeg. Each stage is independent; the middle one is the only one that talks to the API.

Input Source text A chapter from ePub, Markdown, plain text, or a database. Anywhere from 1K to 250K characters.
Stage 1 Chunker Splits on paragraph then sentence boundaries. Hard cap of 20,000 characters per chunk to respect the API limit.
Stage 2 Speechify TTS, looped One POST /v1/audio/speech per chunk, sequential or fanned out with a concurrency bound. Retries with exponential backoff on 429/5xx.
N parallel synthesis calls
part-000.mp3
part-001.mp3
part-002.mp3
part-NNN.mp3
Stage 3 ffmpeg concat Lossless re-mux of every chunk into a single chapter MP3 with -c copy. M4B with chapter markers is the same shape plus a metadata file.
Output chapter-01.mp3 (or .m4b) A single chapter file with consistent voice, no audible chunk seams, and chapter markers if you went the m4b route.
Only stage 2 touches the API. Stages 1 and 3 are pure local processing. Restartability and caching at the chunk boundary fall out of the shape for free: hash each chunk's input text, key cached MP3s on the hash, never re-synthesize a chunk whose source hasn't changed.
